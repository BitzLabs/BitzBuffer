# BitzBuffer.Pipelines 設計仕様 - ネットワーク通信トランスポート

このドキュメントは、BitzBuffer.Pipelines におけるネットワーク通信（主にTCPおよびUDP）を実現するためのトランスポート層の詳細設計について記述します。異なるマシン間で効率的にデータを送受信するための仕組みを提供します。

トランスポート層の共通的な設計方針については、[`03_P_トランスポート層概要.md`](./03_P_トランスポート層概要.md) を参照してください。
コアパイプライン抽象については [`01_P_コアパイプライン抽象.md`](./01_P_コアパイプライン抽象.md) を参照してください。

## 1. はじめに

ネットワーク通信は、分散システムやクライアントサーバーアプリケーションの根幹をなす技術です。BitzBuffer.Pipelines は、BitzBufferコアの効率的なバッファ管理と、ノンブロッキングなパイプライン処理を組み合わせることで、高性能かつスケーラブルなネットワーク通信機能を提供することを目指します。

このドキュメントでは、TCP (ストリームベース) および UDP (データグラムベース) プロトコルに対応するトランスポート実装の設計方針を検討します。

## 2. ネットワークトランスポートの共通要件と課題

*   **ソケットAPIの利用:** .NET の `System.Net.Sockets.Socket` クラスを中心としたAPIを利用。
*   **非同期I/O:** `SocketAsyncEventArgs` や `Socket` の非同期メソッド (`ReceiveAsync`, `SendAsync`, `ConnectAsync`, `AcceptAsync`) を最大限に活用。
*   **接続管理 (TCP):** 接続の確立、維持、切断、タイムアウト処理。
*   **アドレス解決:** ホスト名からIPアドレスへの解決。
*   **データのフレーミング (TCP):** ストリーム指向のTCPでメッセージ境界を扱うための仕組み（例: 長さプレフィックス、区切り文字）。
*   **エラーハンドリング:** ネットワークエラー、接続エラー、タイムアウトなどの適切な処理。
*   **セキュリティ (オプション):** TLS/SSLによる暗号化通信のサポート（初期実装ではスコープ外とするか、基本的なフックポイントのみ提供）。
*   **スケーラビリティ:** 多数の同時接続を効率的に処理できる設計（特にサーバー側）。
*   **シリアライズ/デシリアライズ:** アプリケーションデータをネットワーク送受信可能なバイト列に変換。

## 3. TCPトランスポート

信頼性のある順序付けられたストリーム通信を提供します。

### 3.1. 設計方針

*   `IPipeConnection` および `IPipeConnectionListener` インターフェース（またはそれに類するネットワーク用抽象）を実装。
*   内部で `Socket` (TCP用) をラップし、その非同期送受信処理と `BitzPipe<byte>` を連携させる。
*   データはバイトストリームとして扱う。メッセージ境界の処理は、このトランスポート層または上位のプロトコル層の責務。

### 3.2. APIとオプション (案)

*   **`TcpConnectionListenerOptions`:**
    *   `IPEndPoint LocalEndPoint`: リッスンするローカルIPアドレスとポート。
    *   `int ListenBacklog`: 保留可能な接続要求のキューの長さ。
    *   ソケットオプション (`NoDelay`, `LingerState` など)。
*   **`TcpConnectionOptions` (クライアント側およびサーバーがAcceptした接続用):**
    *   `EndPoint RemoteEndPoint`: 接続先のリモートエンドポイント (クライアント時)。
    *   ソケットオプション。
    *   キープアライブ設定。
*   **提供するクラス (案):**
    *   `TcpBitzConnectionListener : IPipeConnectionListener`
    *   `TcpBitzConnection : IPipeConnection`

### 3.3. データフレーミング戦略

TCPはストリーム指向であるため、メッセージの境界を定義するためのフレーミング戦略が不可欠です。

*   **長さプレフィックス:** 各メッセージの先頭にメッセージ長を付加する。
*   **区切り文字:** メッセージの末尾に特定の区切り文字シーケンスを付加する。
*   BitzBuffer.Pipelines のTCPトランスポートは、これらの一般的なフレーミング戦略を選択可能にするか、あるいはフレーミングは利用者の責務とし、トランスポートは生のバイトストリームのみを扱うか、検討が必要です。初期実装では、生のバイトストリームを基本とし、フレーミングは上位層の責務とするのがシンプルかもしれません。

## 4. UDPトランスポート

コネクションレスで、信頼性よりも低遅延が重視されるデータグラム通信を提供します。

### 4.1. 設計方針

*   `IDatagramPipe` インターフェース（またはそれに類する抽象）を実装。
*   内部で `Socket` (UDP用) をラップし、非同期送受信 (`ReceiveFromAsync`, `SendToAsync`) と `BitzPipe<DatagramMessage>` を連携させる。
*   `DatagramMessage` は、ペイロード (`ReadOnlyMemory<byte>`) と送信元/送信先 `EndPoint` を保持する構造体。

### 4.2. APIとオプション (案)

*   **`UdpPipeOptions`:**
    *   `IPEndPoint? LocalEndPoint`: バインドするローカルエンドポイント（オプション、指定しない場合はOSが割り当て）。
    *   ソケットオプション (`EnableBroadcast`, `MulticastLoopback` など)。
    *   受信バッファサイズ。
*   **提供するクラス (案):**
    *   `UdpBitzPipe : IDatagramPipe`

### 4.3. メッセージサイズと順序、信頼性

UDPはメッセージの順序や到達を保証しません。大きなデータはMTUサイズに合わせて分割する必要があり、信頼性が必要な場合はアプリケーション層で再送制御や順序制御を実装する必要があります。
BitzBuffer.Pipelines のUDPトランスポートは、これらのUDPの特性をそのまま反映し、基本的なデータグラムの送受信機能を提供することに注力します。高レベルな信頼性機能はスコープ外（または将来の拡張）とします。

## 5. シリアライズ/デシリアライズ

ネットワークトランスポートも基本的にバイト列 (`byte`) を扱います。アプリケーションデータのシリアライズ/デシリアライズは、トランスポート層の利用者（または上位のプロトコル層）の責任となります。
(詳細は [`07_P_通信パターン.md`](./07_P_通信パターン.md) や将来のシリアライズ関連ドキュメントで検討)

## 6. 将来の拡張 (ネットワークトランスポート関連)

*   **TLS/SSLサポート:** TCPトランスポートにおける暗号化通信の統合。
*   **HTTP/WebSocketトランスポート:** より高レベルなWebプロトコルへの対応。
*   **QUICトランスポート:** UDPベースの新しいトランスポートプロトコルへの対応。
*   **マルチキャスト/ブロードキャストの高度なサポート (UDP)。**
*   **ネットワーク状態変化への対応強化** (接続断の検知と再接続ロジックのヘルパーなど)。
