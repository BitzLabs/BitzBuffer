# BitzBuffer.Pipelines 設計仕様 - 将来の拡張

このドキュメントは、BitzBuffer.Pipelines ライブラリの初期実装ではスコープ外とするものの、将来的に検討または実装される可能性のある拡張機能や改善点をリストアップします。これにより、ライブラリの発展的なロードマップと、現時点での設計が考慮している将来性を示します。

各機能コンポーネント（コアパイプライン抽象、各トランスポート、通信パターンなど）の詳細な設計仕様書にも、それぞれ固有の将来の拡張項目が記載されていますが、このファイルではより横断的または大規模な拡張アイデアを中心に記述します。

## 1. コアパイプライン機能の強化

*   **より高度な `PipeScheduler` の提供:**
    *   優先度付きスケジューリング。
    *   特定のスレッドへのアフィニティ指定。
    *   カスタム同期コンテキストとのより深い統合。
*   **`BitzPipeReader<T>` / `BitzPipeWriter<T>` のヘルパーメソッド拡充:**
    *   `System.IO.Pipelines` に存在するような、より多くの便利なヘルパーメソッド（例: `ReadExactlyAsync`, `ReadAtLeastAsync`, `WriteMessagesAsync` [特定のデリミタやフォーマットで] など）の追加。
    *   `CopyToAsync(Stream destination)` や `Stream.ReadAsync(PipeWriter<byte> destination)` といった `System.IO.Stream` との高度な連携機能。
*   **パイプの検査とデバッグ機能:**
    *   パイプラインの内部状態（バッファリング量、リーダー/ライターの状態など）を外部から検査するためのAPI。
    *   デバッグ時の診断情報出力の強化。
*   **マルチリーダー/マルチライターパイプのサポート:**
    *   現在のシングルリーダー/シングルライターモデルに加え、複数のリーダーが同じデータを消費できるブロードキャスト型パイプや、複数のライターが一つのパイプに書き込めるマージ型パイプのサポートを検討。これには高度な同期とデータ管理が必要。

## 2. トランスポート層の拡張

*   **新しい通信プロトコル/技術への対応:**
    *   QUIC トランスポート。
    *   WebSockets トランスポート。
    *   gRPC トランスポートの基盤としての利用。
    *   より多くの種類のIPCメカニズム（メッセージキューなど）。
*   **トランスポートレベルでのセキュリティ機能:**
    *   TLS/SSL設定オプションの拡充と、より容易な統合。
    *   IPCにおけるセキュリティ属性やアクセス制御の強化。
*   **接続管理の高度化:**
    *   自動再接続ロジックの組み込み（オプション）。
    *   コネクションプーリング（特にクライアント側）。
    *   ヘルスチェックと不健全な接続の自動切断/再確立。
*   **クロスプラットフォーム互換性の向上:**
    *   各トランスポートが、主要なオペレーティングシステム（Windows, Linux, macOS）で一貫して動作するように、プラットフォーム固有の挙動を吸収。

## 3. 高レベル通信パターンの拡充と強化

*   **PUB/SUB:**
    *   メッセージフィルタリング機能の強化。
    *   様々な配信保証レベル（At-most-once, At-least-once, Exactly-once）のサポート検討。
    *   メッセージの永続化オプション。
    *   デッドレターキュー (DLQ) のサポート。
*   **Request/Response (RPC):**
    *   インターフェース定義言語 (IDL) やコード生成ツールとの連携。
    *   サービスディスカバリ機構との統合。
    *   クライアントサイドロードバランシング。
    *   より詳細なエラー伝播メカニズム。
    *   ストリーミングRPCのサポート。
*   **新しい通信パターンの追加:**
    *   ワークキューパターン。
    *   MapReduceライクな分散処理パターン。
    *   アクターモデルとの親和性を高める機能。

## 4. WebGPU パイプライン連携の詳細設計

*   [`../BitzBuffer/04_GPUサポート.md`](../BitzBuffer/04_GPUサポート.md) で構想されているBitzBufferコアのGPUサポートと連携し、CPU側で準備されたデータをGPU処理パイプラインへ効率的に供給したり、GPUからの結果を後続のCPUパイプラインへ流したりするための具体的なAPIと実装を設計します。
*   これには、GPUメモリ（`IBuffer<T>`のGPU実装）を直接扱える特殊なパイプステージや、GPUコマンドキューとの同期を考慮したデータ転送メカニズムが含まれる可能性があります。

## 5. パフォーマンスと診断

*   **パフォーマンス最適化:**
    *   ロックフリーデータ構造やアルゴリズムの導入検討（プロファイリングに基づく）。
    *   アロケーションのさらなる削減。
    *   各トランスポートのオーバーヘッド削減。
*   **診断機能の強化:**
    *   分散トレーシング (OpenTelemetryなど) との統合。
    *   より詳細なパフォーマンスカウンターやメトリクスの公開。

## 6. その他

*   **設定ファイルからのパイプライン構成読み込み:** アプリケーションコードを変更せずに、パイプラインのトポロジーやオプションを設定ファイルから構築できる機能。
*   **リアクティブ拡張 (Rx.NET) との統合:** `IObservable<T>` / `IAsyncEnumerable<T>` と `BitzPipeReader<T>` / `BitzPipeWriter<T>` との相互変換や連携を容易にするアダプタの提供。
