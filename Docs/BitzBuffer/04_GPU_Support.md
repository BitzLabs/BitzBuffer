# BitzBuffer 設計仕様 - GPUサポート

このドキュメントは、C# Buffer管理ライブラリ「BitzBuffer」が将来的にGPU (Graphics Processing Unit) バッファの管理をサポートするための拡張アーキテクチャと設計方針について記述します。初期実装ではGPUサポートは直接含まれませんが、コアライブラリの設計はGPU対応を容易にするための拡張性を考慮します。

GPUサポートは、特定のGPU API (Vulkan, OpenGL, DirectX, Metalなど) に依存するため、通常は別個の拡張ライブラリ (DLL) として提供されることを想定しています。

## 8. GPUサポートの実現（拡張アーキテクチャ経由）

本ライブラリのコアコンポーネント (`IBuffer<T>`, `IBufferProvider`, `BufferManager` など) は、特定のGPU APIに直接依存しません。GPUサポートは、これらのコアインターフェースを実装・拡張する形で実現されます。

### 8.1. CPU向け設計におけるGPUサポートのための基盤

BitzBufferのコアライブラリ（CPUメモリ管理機能）は、将来的なGPUサポートを円滑に導入できるよう、以下の設計原則と抽象化を考慮しています。これらがGPUプロバイダとGPUバッファ実装の基盤となります。

*   **インターフェースによる抽象化**
*   **厳格なリソースライフサイクル管理**
*   **メモリ特性の柔軟な指定**
*   **基本的なメモリアクセスプリミティブの重視**
*   **同期に関する責任分担**

これらのCPU側での設計配慮により、GPUサポート拡張ライブラリは、BitzBufferのコアフレームワークにスムーズに統合されることが期待されます。

### 8.2. GPUプロバイダ (`IBufferProvider`の実装)

*   **役割:** 各GPU APIに対応する専用の `IBufferProvider` の実装クラスを作成。
*   **責務:** API固有のバッファ確保・解放・メモリ管理、デバイス選択、プーリング戦略の実装。
*   **登録と利用:** `BufferManager` に登録し、名前で取得して利用。
*   **`IDisposable` の実装:** GPUコンテキストやリソースを解放。

### 8.3. GPUバッファの実装クラス (`IBuffer<T>`の実装)

*   **役割:** 各GPUプロバイダが提供する `IBuffer<T>` の実装。
*   **内部構造:** ネイティブリソースハンドルを `SafeHandle` でラップ。GPU固有情報を保持。
*   **所有権とライフサイクル:** コアライブラリのルールに従う (`IBufferState`, `IOwnedResource`)。
*   **GPU固有インターフェース:** `IMappableGpuBuffer<T>` など、追加機能を提供。
*   **`ToString()`:** GPU固有情報を含むデバッグ情報。

### 8.4. GPUバッファのオプションと設定

*   **`ProviderOptionsBuilder` の拡張:** GPUプロバイダ固有のオプション設定。（詳細は [`./02_Providers_And_Buffers.md`](./02_Providers_And_Buffers.md) 参照）
*   **設定項目例:** デバイス選択、メモリタイプ、用途、プーリングパラメータなど。
*   **`Rent`/`CreateBuffer` 時のオプション:** 型安全なオプションオブジェクトで用途や可視性を指定。

### 8.5. GPUリソースのプーリングとライフサイクル管理

*   **プーリング戦略:** コアライブラリの仕組み (`IBufferPoolStrategy`) を適用。（詳細は [`./03_Pooling.md`](./03_Pooling.md) 参照）
*   **ライフサイクルフック (`IBufferLifecycleHooks`):** GPUバッファ固有の状態リセットや解放処理。
*   **アライメントとオフセット:** GPUの厳密な要件を考慮。

### 8.6. データ転送 (CPU と GPU 間)

*   **役割分担:** コアは直接的な転送コマンドを持たず、GPUプロバイダや利用者が担当。
*   **ステージングバッファ:** `IBuffer<T>` として管理。
*   **`TryAttachZeroCopy` / `AttachSequence` の応用:** ステージングバッファへのマッピングなど。
*   **マッピング:** `IMappableGpuBuffer<T>.Map()/Unmap()`。

### 8.7. 同期

*   **責任範囲:** アプリケーションコードが主に担当。
*   **ライブラリの配慮:** 外部同期機構との連携を考慮。

### 8.8. 将来の拡張 (GPUサポート関連)

*   **対応GPU APIの拡充:** DirectX, Metal など。
*   **GPUリソース特化プーリング戦略**
*   **高度なデータ転送ユーティリティ**
*   **同期プリミティブとの連携強化**
*   **コンピュートシェーダ連携の最適化**
