# 02_ブランチ戦略と命名規則

このドキュメントでは、本ライブラリ開発におけるGitのブランチ戦略と命名規則について説明します。
一人開発であっても、一貫したブランチ戦略を持つことで、開発プロセスの整理、変更履歴の追跡、そして将来の自分自身のための明確な記録を残すことができます。

## 1. 基本方針

本プロジェクトでは、シンプルで柔軟な **GitHub Flow** をベースとしたブランチ戦略を採用します。

*   **`main` ブランチ:**
    *   常に安定しており、リリース可能な状態（または最新の安定版）を保ちます。
    *   直接このブランチにコミットすることは避け、プルリクエスト (PR) 経由でのみ変更をマージします。

## 2. ブランチの種類と命名規則

作業の種類に応じて、以下の命名規則でブランチを作成します。ブランチ名には、関連するIssue番号を含めることを強く推奨します。これにより、ブランチの目的が明確になり、Issueとの関連付けも容易になります。

### 2.1. `main` ブランチ

*   **名前:** `main`
*   **役割:** プロジェクトの主軸となる安定したブランチ。タグ付けによるバージョンリリースの基点となります。

### 2.2. フィーチャーブランチ (Feature Branches)

新しい機能を追加したり、既存機能を拡張したりする際に使用します。

*   **命名規則:**
    *   `feat/<Issue番号>-<簡単な説明>`
*   **例:**
    *   `feat/12-add-user-authentication`
*   **派生元:** `main`
*   **マージ先:** `main` (プルリクエスト経由)

### 2.3. バグ修正ブランチ (Bugfix Branches)

バグを修正する際に使用します。

*   **命名規則:**
    *   `fix/<Issue番号>-<簡単な説明>`
*   **例:**
    *   `fix/42-resolve-login-page-error`
*   **派生元:** `main`
*   **マージ先:** `main` (プルリクエスト経由)

### 2.4. リファクタリングブランチ (Refactor Branches)

コードの内部構造を改善し、可読性やパフォーマンスを向上させる際に使用します。外部から見た動作は変更しません。

*   **命名規則:**
    *   `refact/<Issue番号>-<簡単な説明>`
*   **例:**
    *   `refact/61-optimize-database-query`
*   **派生元:** `main`
*   **マージ先:** `main` (プルリクエスト経由)

### 2.5. ドキュメントブランチ (Documentation Branches)

READMEの更新や、その他のドキュメント作成・修正を行う際に使用します。

*   **命名規則:**
    *   `docs/<Issue番号>-<簡単な説明>`
*   **例:**
    *   `docs/92-add-contribution-guidelines`
*   **派生元:** `main`
*   **マージ先:** `main` (プルリクエスト経由)

### 2.6. その他タスクブランチ (Task/Chore Branches)

上記に当てはまらない一般的なタスクや、ビルド設定の変更、ライブラリのアップデートなどを行う際に使用します。

*   **命名規則:**
    *   `task/<Issue番号>-<簡単な説明>`
*   **例:**
    *   `task/101-setup-ci-workflow`
*   **派生元:** `main`
*   **マージ先:** `main` (プルリクエスト経由)

## 3. 基本的なワークフロー

1.  **Issueの作成 (推奨):**
    *   これから行う作業に対応するIssueをGitHub上で作成します。
    *   Issueには適切なラベル（`bug`, `feat`, `task`など）を付与します。

2.  **`main` ブランチの最新化:**
    *   ローカルの `main` ブランチを最新の状態にします。
      ```bash
      git checkout main
      git pull origin main
      ```

3.  **作業ブランチの作成:**
    *   GitHubのIssueページから「Create a branch」機能を利用するか、ローカルで上記の命名規則に従ってブランチを作成します。
      ```bash
      # 例: フィーチャーブランチの作成
      git checkout -b feat/123-my-new-feature main
      ```

4.  **ローカルでの作業とコミット:**
    *   新しいブランチでコードの変更、テストの作成・実行を行います。
    *   意味のある単位で、分かりやすいメッセージと共にコミットします。コミットメッセージに `Closes #<Issue番号>` や `Fixes #<Issue番号>` を含めると、PRマージ時にIssueを自動でクローズできます。
      ```bash
      git add .
      git commit -m "feat: 新機能のコアロジックを実装 (Closes #123)"
      ```

5.  **リモートへのプッシュ:**
    *   定期的に作業内容をリモートの同名ブランチにプッシュします。
      ```bash
      git push -u origin feat/123-my-new-feature
      ```

6.  **プルリクエスト (PR) の作成:**
    *   作業がある程度完了した時点、またはレビューを受けたい時点で、GitHub上で作業ブランチから `main` ブランチへのプルリクエストを作成します。
    *   PRのタイトルや説明には、変更の目的、内容、関連するIssue番号を明記します。

7.  **セルフレビュー:**
    *   作成したPRの「Files changed」タブで、自身の変更内容を客観的にレビューします。
    *   コードの品質、テストの網羅性、ドキュメントの必要性などを確認します。

8.  **自動テストの確認 (もしあれば):**
    *   GitHub ActionsなどでCIが設定されていれば、ビルドとテストの結果を確認します。

9.  **プルリクエストのマージ:**
    *   セルフレビューと自動テストに問題がなければ、プルリクエストを `main` ブランチにマージします。
    *   GitHub上の「Merge pull request」ボタンを使用します。

10. **マージ後のブランチ削除:**
    *   `main` へのマージ後、作業ブランチは不要になるため削除します (GitHubが削除ボタンを提案してくれます)。
    *   ローカルのブランチも不要であれば削除します。
      ```bash
      git checkout main
      git branch -d feat/123-my-new-feature
      ```

11. **(任意) タグ付けとリリース:**
    *   `main` ブランチが更新されたら、必要に応じてバージョンタグを打ち、リリース準備を行います。
      ```bash
      git tag v1.0.0
      git push origin v1.0.0
      ```

## 4. プルリクエスト (PR) の役割

一人開発であっても、PRを作成することには以下のメリットがあります。

*   **変更履歴の明確化:** 各変更の目的、議論の経緯（セルフレビューのコメントなど）が記録として残ります。
*   **セルフレビューの機会:** 自分のコードを客観的に見直すことで、バグの早期発見や品質向上に繋がります。
*   **CI/CDのトリガー:** 自動テストやビルド、デプロイなどの処理をPR作成時やマージ時に自動実行できます。

## 5. コミットメッセージの指針

*   **簡潔かつ具体的に:** コミットが何を変更したのかが一目で分かるように記述します。
*   **現在形・命令形で:** `Fixes bug` や `Add feature` のように記述します。(英語の場合)
*   **関連Issueの記述:** 可能であれば、`Closes #<Issue番号>` や `Refs #<Issue番号>` のように関連するIssue番号を記述します。

## 6. まとめ

このブランチ戦略と命名規則に従うことで、開発プロセスが整理され、変更の追跡が容易になります。
状況に応じて柔軟に運用しつつ、一貫性を保つことを目指しましょう。
